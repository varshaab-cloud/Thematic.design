## AlertDialog copy — the three-register rule

The copy in a confirmation dialog determines whether users feel in control or manipulated. There is one right register for enterprise software.

    { el: "Body", rule: "State the exact consequence in one sentence. Add 'This can't be undone.' as a second sentence if and only if it's true.", example: "\"She'll lose access to all 12 projects immediately. This can't be undone.\"" },

## Destructive button — when and where

The `destructive` variant of Button exists for a specific job: the final confirm action inside an AlertDialog, or an inline action that initiates a destructive flow. It is not for general warnings.

## Undo as an alternative to confirmation

For Level 1 actions, a Toast with an undo action is always better than a confirmation dialog. Confirmation dialogs add friction before the action. Undo removes friction and lets users act fast — with a safety net.

  <div style={{ display: "flex", alignItems: "center", justifyContent: "space-between", background: "#222", borderRadius: 8, padding: "12px 16px", marginBottom: 12 }}>
    <span style={{ fontSize: 13, color: "white" }}>Priya Sharma removed from workspace.</span>
    <span style={{ fontSize: 12, fontWeight: 600, color: "#A5B4FC", cursor: "pointer", marginLeft: 16 }}>Undo</span>
  </div>
  <p style={{ fontSize: 12, color: "var(--base-color-gray-500, #909090)", margin: 0, lineHeight: 1.5 }}>
    Example: <code>toast("Priya Sharma removed.", {"{ action: { label: 'Undo', onClick: handleUndo }, duration: 6000 }"})</code>
    — The toast persists for 6 seconds (enough time to react), then auto-dismisses. If the user clicks Undo, the action is reversed and a success toast confirms it.
