## Spatial layout rules

<div style={{ display: "grid", gridTemplateColumns: "1fr 1fr", gap: 16 }}>

<div style={{ padding: "20px 20px 16px", border: "1px solid var(--base-color-green-200, #BBF7D0)", borderRadius: 10, background: "var(--base-color-green-100, #DCFCE7)" }}>
  <p style={{ fontSize: 13, fontWeight: 600, color: "var(--base-color-green-800, #166534)", margin: "0 0 10px" }}>✓ Do</p>
  <ul style={{ fontSize: 13, lineHeight: 1.8, color: "var(--base-color-green-800, #166534)", margin: 0, paddingLeft: 18 }}>
    <li>Group related fields inside <code>FormSection</code> with a title</li>
    <li>Use <code>FormRow</code> for pairs: first name + last name, city + postcode</li>
    <li>Keep each section to 3–5 fields max before starting a new section</li>
    <li>Put <code>FormActions</code> at the very end, never mid-form</li>
    <li>Always include a cancel action alongside submit</li>
    <li>Label every field — no placeholder-only inputs in enterprise forms</li>
  </ul>
</div>

<div style={{ padding: "20px 20px 16px", border: "1px solid var(--base-color-red-200, #FFB9BB)", borderRadius: 10, background: "var(--base-color-red-100, #FFE2E2)" }}>
  <p style={{ fontSize: 13, fontWeight: 600, color: "var(--base-color-red-600, #D21A26)", margin: "0 0 10px" }}>✗ Don't</p>
  <ul style={{ fontSize: 13, lineHeight: 1.8, color: "var(--base-color-red-600, #D21A26)", margin: 0, paddingLeft: 18 }}>
    <li>Don't put more than 2 fields in a <code>FormRow</code></li>
    <li>Don't mix wide and narrow fields in the same row</li>
    <li>Don't skip <code>FormItem</code> wrapper — it carries the a11y context</li>
    <li>Don't suppress error messages — always surface <code>fieldState.error?.message</code></li>
    <li>Don't use required asterisk without also communicating it in the schema</li>
    <li>Don't nest <code>FormSection</code> components</li>

## Error message copy

Error messages are the most-read copy in a form. They must follow three rules: **diagnose** what happened, **explain** why, and **recover** — tell the user what to do next.

## Destructive action confirmations

Use `AlertDialog` for any action that is irreversible or removes data permanently. The copy pattern determines whether users feel in control or alarmed.

The right register for enterprise: **direct**, **specific**, and **respectful of the user's intelligence**. State exactly what will be deleted or lost. Don't soften ("remove" not "delete"), don't alarm, don't use vague warnings.

<div style={{ display: "grid", gridTemplateColumns: "1fr", gap: 16, marginTop: 24, marginBottom: 24 }}>
{[
  { label: "✗ Too clinical", color: "#FFE2E2", textColor: "#D21A26", borderColor: "#FFB9BB",
    title: "Confirm deletion?",
    body: "Are you sure you want to delete this item? This cannot be undone.",
    cancel: "Cancel", confirm: "Delete" },
  { label: "✗ Too alarming", color: "#FEF9C3", textColor: "#A16207", borderColor: "#FDE047",
    title: "⚠ WARNING",
    body: "This action is PERMANENT and CANNOT be reversed! All data will be lost forever!",
    cancel: "Cancel", confirm: "I understand, delete it" },
  { label: "✓ Right register", color: "#DCFCE7", textColor: "#166534", borderColor: "#BBF7D0",
    title: "Remove Priya Sharma from this workspace?",

      <p style={{ fontSize: 13, color: "#6D6D6D", margin: "0 0 16px", lineHeight: 1.5 }}>{body}</p>
      <div style={{ display: "flex", gap: 8, justifyContent: "flex-end" }}>
        <span style={{ fontSize: 12, padding: "6px 12px", borderRadius: 6, border: "1px solid #E3E3E0", color: "#464646" }}>{cancel}</span>
        <span style={{ fontSize: 12, padding: "6px 12px", borderRadius: 6, background: "#FF4E2A", color: "white" }}>{confirm}</span>

    </div>
  </div>

**Copy rules for destructive confirmations:**
- **Title**: name the specific thing being removed, not the action — "Remove Priya Sharma?" not "Confirm removal"
- **Body**: state the exact consequence + "can't be undone" — no exclamation marks, no ALL CAPS
- **Confirm button**: specific verb + noun — "Remove member", "Delete workspace", "Cancel subscription" — never just "Delete" or "Confirm"
- **Cancel button**: always "Cancel" — nothing more

## Empty states

Every data-driven surface — tables, lists, search results, dashboards — needs a defined empty state. Without one, each engineer builds their own.

**Anatomy of an empty state:**
- **48px icon** — outlined, single colour (gray-500 at rest, blue-800 for CTA-bearing states)
- **Heading** — what's missing, in plain language. Not "No data found."
- **Supporting sentence** — why it's empty or what the user can do about it
- **Optional CTA** — only when there's a clear next action from this page

    {cta && <span style={{ marginTop: 8, fontSize: 12, padding: "6px 14px", borderRadius: 8, background: "#1518A6", color: "white" }}>{cta}</span>}
  </div>
))}

</div>

**Copy rules for empty states:**
- **Heading**: specific — name what's missing. "No team members yet", not "Nothing here."
- **Body**: explain why it's empty or what action creates data here.
- **CTA**: only include if there's a direct action. "No results" empty states don't need a CTA.
- Never say "No data found" — it's system language, not user language.
